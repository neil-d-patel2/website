Lineup Optimization:
A Markov Chain Approach to MLB Batting Orders

Baserunner-Dependent Net Run Production · A full-stack optimization system for context-aware batting-order decisions

JOHNS HOPKINS UNIVERSITY
Department of Computer Science · 2026
Next.jsFastAPIPython Numba JITPostgreSQL MLB Stats APIVercel · Fly.io
Abstract
We present a full-stack lineup optimization system that models an MLB batting order as a Markov chain over twenty-four base-out states. Each batter's contribution is quantified through Baserunner-Dependent Net Run Production (BRP) — a four-batter window metric that captures the context-dependent value missed by traditional statistics such as OPS or wRC+. An exhaustive search over all 9! = 362,880 permutations (accelerated with Numba JIT, under two seconds) identifies the optimal batting order, which is then compared against actual manager decisions across the full MLB schedule.

Starting from bases empty, zero outs, each batter expands into every possible base-out state, weighted by their transition probabilities. Runner advancement depends on the speed factors of preceding batters.

T[i, j, k, l] = BRP(batteri, batterj, batterk, batterl) → A 9 × 9 × 9 × 9 BDNRP tensor

The tensor encodes the expected net run production for every ordered quadruple of batters, forming the search space for optimization.

3BRP FORMULA

Run Expectancy Valuation

Each terminal base-out state is valued using the RE24 matrix, which gives the expected runs from any state through the end of the inning.

BRPstate = (runs + RE24[new] − RE24[start]) × P(state) total_BRP = Σ BRPstate over all terminal states

PART II
Optimization Engine
4SCORING & EXHAUSTIVE SEARCH

Position-Weighted BRP Across the Order

The lineup score uses a sliding window of four that wraps around the order (position 9 feeds into positions 1–3), capturing the cyclic nature of a batting order.

score = Σ T[order[(p−3) mod 9], order[(p−2) mod 9], order[(p−1) mod 9], order[p]] × w[p] w[p] = 1 + (8 − p) / 9 for p = 0 … 7 w[8] = 1.0 9th spot

Supported constraints: positional locks (pin a player to a slot), handedness limits (maximum consecutive LHB or RHB, with wraparound), and speed-based platoon splits.

5STAT BLENDING

Bayesian Regression to the League Average

Raw rates from small samples produce unreliable BRP values. We blend observed stats with prior-season performance and 50 phantom plate appearances at league-average rates:

rate = (obs + prior + 50 × avg) / (PAobs + PAprior + 50\) output = round(rate × 600 PA) normalizes every player to 600 PA

Platoon splits: vs-LHP and vs-RHP stats are pulled from the current and prior season. The pipeline falls back to overall rates whenever split PA equals zero.

10Engineering Challenges

Bugs Fixed & Solutions

!
Negative PUDI bug. Single-game samples (four PA) produced unreliable BRP. Fixed by fetching and blending prior-season counting stats before regression.
!
Regression flattening. A value of REGRESS_PA = 200 compressed player differences to deltas of roughly 0.01. Reducing it to 50 preserves real variation while still stabilizing small samples.
!
Apples-to-oranges BRP. The actual and optimized panels used different stat inputs. Resolved with a dual-call architecture in which the optimize call also returns the actual lineup's BRP using identical blended stats.
!
MLB API split-season bug. The season parameter must live inside the hydrate string rather than as a top-level query parameter. The original code silently fetched current-year splits for every requested season.
